# SITE_IMPROVEMENT_AUDIT

작성일: 2026-06-11

감사 기준: 코드, 기존 문서, 테스트 구조, 대표 라우트 구성 기반 정적 점검. 실제 업무 파일과 기준 원가표가 제공되지 않은 항목은 `추정` 또는 `추가 확인 필요`로 표시했다.

수정 원칙: 이 문서는 개선 후보와 QA 항목을 정리한 문서이며, production 코드, 계산식, 저장/수정/삭제, 백업/복원, 업로드/다운로드, 엑셀 출력 형식, API 응답 구조, 라우터 구조를 변경하지 않는다.

## 1. 전체 요약

현재 사이트는 메뉴 판매량, 제때 단가/출고, 식자재, 메뉴 마스터, 원가, 영양성분, 노트, 보고서, 설정/백업/복원까지 업무 범위가 넓고, 대부분의 데이터가 브라우저 IndexedDB와 파일 입출력 흐름에 의존한다. `docs/FEATURE_MAP.md`, `docs/QA_CHECKLIST.md`, `docs/REGRESSION_TEST_PLAN.md`, `scripts/smoke-qa.mjs`, `__tests__` 구조를 보면 기본 화면 순회와 helper 단위 테스트는 상당히 준비되어 있지만, 실제 업무 파일과 기준 원가표를 사용한 숫자 검증은 사람이 반드시 확인해야 한다.

- 데이터 정확성: 원가, 판매가, 원가율, 제품코드, 메뉴코드, 엑셀 출력값은 업무 의사결정에 직접 영향을 준다. 실데이터 기준표와 Excel 앱 확인이 없는 상태에서는 자동 테스트만으로 안전하다고 판단하기 어렵다.
- 안정성: IndexedDB 스키마, 백업/복원, 삭제/숨김, 중복 업로드 차단은 운영 데이터 보존과 직결된다. 복구 가능한 QA 데이터와 사전 백업을 전제로 검증해야 한다.
- 속도: `xlsx`는 lazy load 구조가 확인되지만, 대량 테이블 렌더링, 대용량 엑셀 파싱, IndexedDB 전수 조회, 홈 대시보드 통계 집계는 데이터가 많아질수록 체감 지연 가능성이 있다.
- 사용성: 검색/필터/정렬/페이지네이션, 업로드 오류 행 표시, 저장 중 피드백, 삭제 경고, 모바일 overflow 확인은 실사용 안정감을 크게 좌우한다.
- 코드 품질: 공통 UI와 guard 테스트가 존재하고 모듈 분리도 진행되어 있다. 다만 업무 fixture, 계산 snapshot, 엑셀 workbook 구조 테스트, 대량 데이터 성능 기준은 추가 보강 후보로 남아 있다.

핵심 보완 방향은 “많이 고치기”가 아니라, 실사용 전에 A급 업무 리스크를 기준 파일로 검증하고, 이후 B급 사용성/속도/테스트 보강, 마지막으로 C급 모션/디자인/자동화를 확장하는 순서가 적절하다.

## 2. 우선순위 요약

| 우선순위 | 기준 | 대표 보완 항목 | 권장 대응 |
| --- | --- | --- | --- |
| A | 실사용 전 확인 필요 | 원가 계산, 엑셀 처리, 저장/삭제/복원, 메뉴코드/제품코드 매칭, 데이터 정확성 | 실제 업무 파일, 기준 원가표, 백업 파일로 수동 QA 후 회귀 테스트 후보화 |
| B | 실사용 품질 향상 | 검색/필터/정렬, 페이지네이션, 화면 정리, 로딩/에러, 테스트 보강 | 코드 수정 전 QA 시나리오 확정, 저위험 UI와 테스트부터 단계 적용 |
| C | 장기 개선 | 모션, 디자인 디테일, 추가 기능, 자동화, 교육/운영 문서 | 업무 안정화 후 선택 적용 |

A급에서 가장 먼저 확인할 항목은 다음이다.

- 대표 메뉴별 원가, 판매가, 원가율이 기준표와 일치하는지 확인
- 메뉴 판매량/제때 단가/원산지/영양성분 엑셀 업로드와 다운로드 결과를 실제 Excel 앱에서 확인
- 메뉴코드와 제품코드가 실데이터 기준으로 중복 없이 매칭되는지 확인
- 저장/수정/삭제/백업/복원을 복구 가능한 테스트 데이터에서 확인
- 중복 업로드, 필수 컬럼 누락, 매칭 실패가 명확히 차단 또는 분리되는지 확인

## 8. 실데이터 QA 체크리스트

### 엑셀 업로드/다운로드

- [ ] 메뉴 판매량 실제 `.xlsx`, `.xls`, `.csv` 업로드가 성공한다.
- [ ] 메뉴 판매량 기간과 헤더가 올바르게 감지된다.
- [ ] 제때 단가 실제 파일 업로드가 성공한다.
- [ ] 메뉴 판매량 20MB, 제때 단가/배송 30MB, 백업 복원 500MB 등 모듈별 파일 크기 제한이 업무 파일 크기와 맞는지 확인한다.
- [ ] 메뉴 판매가 업로드처럼 공통 드롭존을 거치지 않는 파일 선택 경로도 대용량 파일, 잘못된 확장자, 빈 파일을 동일한 수준으로 차단하는지 확인한다.
- [ ] 원산지/영양성분 다운로드 파일이 생성된다.
- [ ] 다운로드 파일의 시트명, 헤더, 한글, 줄바꿈, 숫자 형식이 Excel 앱에서 깨지지 않는다.
- [ ] 다운로드 파일명이 `업무명_YYYYMMDD.ext` 형식으로 생성되는지, 기존 업무 자동화와 파일 정렬 기준에 영향이 없는지 확인한다.
- [ ] 식자재 사용현황 CSV/XLSX 컬럼 추가가 공식 템플릿, 외부 비교표, 담당자 검수표에 영향이 없는지 확인한다.
- [ ] 잘못된 확장자, 빈 파일, 필수 컬럼 누락 파일에서 명확한 오류가 표시된다.

### 저장/수정/삭제/백업/복원

- [ ] 실데이터 QA 전에 설정/백업에서 JSON 백업을 만든다.
- [ ] 메뉴 마스터 신규 저장, 수정, 삭제 또는 비활성 처리를 테스트 데이터로 확인한다.
- [ ] 식자재 등록, 수정, 숨김/삭제 상태를 테스트 데이터로 확인한다.
- [ ] 메뉴 판매량/제때 단가 업로드 이력 삭제를 운영 데이터가 아닌 테스트 데이터에서 확인한다.
- [ ] 정상 백업 파일 복원 후 주요 화면의 데이터 수와 대표 값이 복원 전과 일치한다.
- [ ] 잘못된 백업 파일은 복원 전에 차단된다.
- [ ] 백업 파일명 `7번가시스템백업_YYYYMMDD.json`과 복원 전 임시 백업 파일명이 업무자가 식별 가능한지 확인한다.
- [ ] 식자재 삭제 실행취소 시 식자재 본문뿐 아니라 영양값, 알레르기 링크 같은 cascade 삭제 데이터까지 복구되는지 확인한다.
- [ ] 100MB 이상 백업 JSON을 읽을 때 브라우저 멈춤, 메모리 급증, 복원 진행률 누락이 없는지 별도 테스트 데이터로 확인한다.
- [ ] 영양성분 출력 설정 localStorage가 백업/복원되는지, 위젯/필터/프로필/PIN/검색 이력은 복구 범위인지 정책을 확인한다.

### 검색/필터/정렬/페이지네이션

- [ ] 메뉴명, 메뉴코드 검색이 동작한다.
- [ ] 제품명, 제품코드 검색이 동작한다.
- [ ] 분류, 기간, 상태 필터가 동작한다.
- [ ] 숫자 컬럼과 문자열 컬럼 정렬이 각각 올바르게 동작한다.
- [ ] 첫 페이지, 중간 페이지, 마지막 페이지 이동이 동작한다.
- [ ] 빈 결과에서 페이지네이션과 empty state가 깨지지 않는다.

### 원가 숫자 비교

- [ ] 대표 피자 메뉴 3개 이상을 기준표와 비교한다.
- [ ] 사이드 메뉴 3개 이상을 기준표와 비교한다.
- [ ] 세트/1인피자/엣지/도우 항목을 각각 1개 이상 비교한다.
- [ ] 판매가가 0 또는 비어 있는 항목에서 원가율 표시가 깨지지 않는다.
- [ ] 단가 변경 후 영향받는 메뉴의 원가만 예상 범위로 변경된다.

### 제품코드/메뉴코드 매칭

- [ ] 메뉴코드가 메뉴 마스터의 실제 메뉴와 일치한다.
- [ ] 메뉴코드 중복 입력 또는 중복 업로드가 차단된다.
- [ ] 제품코드가 제때 단가/식자재 마스터와 일치한다.
- [ ] 제품코드 중복 입력 또는 중복 업로드가 차단된다.
- [ ] 판매량 미매칭 메뉴가 미매칭 화면에 표시된다.
- [ ] 조합 식자재의 참조 제품코드 누락 여부를 확인한다.

### 중복 검증, 오류 메시지, 로딩 상태

- [ ] 동일 월 판매량 중복 업로드가 차단된다.
- [ ] 동일 날짜 또는 동일 파일 제때 단가 중복 업로드가 차단된다.
- [ ] 오류 메시지가 사용자가 조치할 수 있는 수준으로 표시된다.
- [ ] 업로드/저장/삭제/복원 중 로딩 상태가 표시된다.
- [ ] 성공 시 toast 또는 상태 메시지가 표시된다.
- [ ] 업로드 드롭존과 파일 선택 버튼을 키보드만으로 조작할 수 있는지 확인한다.
- [ ] 설정 PIN과 로그인 쿠키가 보안 경계가 아니라 로컬 접근 방지 장치라는 안내가 운영 정책과 맞는지 확인한다.

### 모바일/데스크톱, 테이블 overflow

- [ ] 모바일 390px 폭에서 주요 화면에 가로 overflow가 없다.
- [ ] 태블릿 702px 폭에서 smoke 대상 화면에 가로 overflow가 없다.
- [ ] 데스크톱에서 카드, 버튼, 테이블 헤더 정렬이 깨지지 않는다.
- [ ] 모달이 화면 높이를 넘을 때 내부 스크롤 또는 닫기 동작이 유지된다.
- [ ] 긴 메뉴명/제품명/메모가 버튼이나 테이블 셀을 침범하지 않는다.
- [ ] Chrome/Safari에서 인쇄 또는 PDF 저장 시 제목, 파일명, 팝업 차단 안내, 원래 페이지 제목 복원이 정상인지 확인한다.

## 12. 최종 추천

현재 사이트는 이미 화면 범위, QA 문서, 회귀 테스트 후보, smoke 자동화가 어느 정도 갖춰져 있다. 따라서 다음 단계는 기능을 많이 추가하는 것이 아니라, 실제 업무 안정성과 실사용성을 기준으로 “검증 가능한 기준”을 먼저 닫는 방향이 현실적이다.

추천 순서는 다음과 같다.

1. 실제 업무 파일과 기준 원가표로 A급 항목을 먼저 확인한다.
2. 확인된 기준값을 익명화 fixture와 회귀 테스트 후보로 전환한다.
3. 저장/삭제/복원, 업로드/다운로드, 코드 매칭처럼 데이터 영향이 큰 작업은 테스트 데이터와 백업 파일이 있을 때만 검증한다.
4. B급 사용성 개선은 오류 메시지, 로딩 상태, 검색/필터/정렬, overflow처럼 실사용 불편이 큰 항목부터 처리한다.
5. C급 모션과 추가 기능은 A/B 리스크가 줄어든 뒤, 업무 흐름을 방해하지 않는 범위에서 선택 적용한다.

최종 판단: 지금 필요한 것은 전면 개편이 아니라, 실데이터 기준의 정확성 검증, 회귀 테스트 보강, 실패/복구 절차 표준화다. 이 순서로 진행하면 업무 신뢰도를 높이면서 불필요한 코드 변경 위험을 줄일 수 있다.

추가 판단: 다운로드 파일명, 출력 컬럼, PDF/인쇄 제목, localStorage 백업 범위는 화면상 작은 변화처럼 보이지만 실제 업무 파일 관리와 복구 절차에 직접 닿는다. 따라서 UI 디테일로 취급하지 말고, 공식 파일명/컬럼/백업 정책을 먼저 확정한 뒤 테스트와 수동 QA 기준으로 고정하는 편이 안전하다.

추가 병렬 점검 판단: 인증/설정 PIN, 업로드 크기 제한, 식자재 삭제 실행취소, 접근성, fixture 문서 상태는 당장 코드를 고치기보다 운영 정책과 실데이터 기준을 먼저 확인해야 한다. 특히 cascade 삭제와 대용량 복원은 작은 UI 문제가 아니라 데이터 손실과 브라우저 안정성에 연결될 수 있으므로 A/B급 검증 항목으로 분리해 추적하는 편이 안전하다.

### 13.4 실데이터로 반드시 확인할 시나리오

- [ ] 기존 백업 또는 현재 업무 DB에 저장된 파생 메뉴가 `ingredientCodes/ingredientAmounts` 없이도 안전하게 표시되는지 확인한다.
- [ ] 새 파생 메뉴를 만들고 L/R 추가 식자재 g 값을 다르게 입력했을 때 계산 결과가 기준표와 일치하는지 확인한다.
- [ ] `씬바사삭R` 행이 실제 정책상 제외되어야 하는지 담당자가 승인한다.
- [ ] 추가토핑 마스터에서 식자재코드 연결, 식자재명 fallback, 알레르기 표시가 모두 맞는지 확인한다.
- [ ] 영양성분 라벨/피자 조각표/추가토핑표/원산지 출력에서 새 토핑과 크러스트 정책이 누락 없이 반영되는지 Excel 또는 인쇄 미리보기로 확인한다.
- [ ] 식자재 사진 3종을 등록한 뒤 목록/관리/백업/복원에서 모두 보존되는지 확인한다.
- [ ] 레시피 식자재를 드래그로 정렬한 뒤 사용량, 단가, 소계, 저장 결과가 행과 함께 유지되는지 확인한다.
- [ ] 식자재 사용현황 CSV의 전체/피자/사이드 메뉴수와 화면 표시값이 실제 메뉴코드 기준 검수표와 일치하는지 확인한다.
- [ ] 모든 주요 다운로드/인쇄 산출물의 파일명, 제목, 컬럼 순서가 담당자 기대와 일치하는지 확인한다.

### 13.6 집중 점검 기준 최종 판단

현재 추가 변경분은 영양성분과 식자재 관리의 업무 범위를 실제로 넓히는 성격이 강하다. 단위 테스트는 일부 보강되어 있지만, 기존 저장 데이터와 실제 출력물까지 보장한다고 보기에는 아직 증거가 부족하다. 따라서 이 변경분은 “기능 구현 완료”보다 “A급 실데이터 QA 대기”로 보는 것이 안전하다.

추가 병렬 점검까지 포함하면, 가장 먼저 확인할 항목은 식자재 삭제 실행취소의 cascade 복구 여부, 업로드/복원 대용량 파일 처리, 인증/PIN의 운영상 의미, 그리고 업로드 접근성이다. 이 항목들은 코드 규모가 크지 않아 보여도 데이터 손실, 복구 실패, 사용 제한으로 이어질 수 있어 실사용 전 확인 가치가 높다.
