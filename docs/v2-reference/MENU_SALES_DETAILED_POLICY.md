## 공통 데이터 정책

### 기본 원칙
- 판매량 집계의 원본은 업로드 완료된 `sales_rows` 데이터다.
- **집계 결과는 DB에 따로 저장하지 않는다.**
- 순위, 비교, 통계 수치는 **화면 조회 시 실시간 계산**한다.
- 원본 엑셀 파일 바이너리는 저장하지 않는다.

### 저장 대상
- 파일 메타데이터
- 파싱된 판매량 행 (`sales_rows`)
- 분류 결과 (`status`, `category`, `groupName`, `detailName`)
- 미매칭 이슈 (`menu_sales_issues`)
- 업로드 로그 (`upload_log`)

### 통계 집계 포함 기준

| 상태 | 포함 | 이유 |
|------|------|------|
| classified | ✅ | 정상 분류 완료 |
| unclassified | ❌ | 분류 실패 → 미매칭 관리 필요 |
| excluded | ❌ | 품목 제외 확정 |

즉, **순위·비교·통계 화면에서는 classified 행만 집계한다.**

## 엑셀/CSV 업로드 정책

### 파일 구조 정책

실제 판매량 엑셀 파일 구조:

| 행 | 내용 |
|-----|------|
| 0행 | 보고서 제목 |
| 1행 | 현재기간: `2026-04-01 ~ 2026-04-30` |
| 2행 | 빈 행 |
| 3행 | **헤더** |
| 4행 이후 | 데이터 |

**주의:**
- 기간이 첫 행에 있다고 가정하지 않는다.
- 헤더가 첫 행에 있다고 가정하지 않는다.
- **기간과 헤더는 파일 첫 10행 안에서 탐색한다.**

### 기간 인식 정책

**업로드는 월 전체 기간만 허용한다.**

**허용:**
- `2026-04-01 ~ 2026-04-30`

**차단:**
- `2026-04-01 ~ 2026-04-17` (월 중간)
- `2026-04-05 ~ 2026-04-30` (시작일이 1일 아님)

**기준:**
- 시작일은 반드시 **1일**
- 종료일은 반드시 **해당 월의 말일**
- 시작월과 종료월은 **같아야 함**

### 헤더 인식 정책

**메뉴명 컬럼 허용값:**
- `메뉴명`
- `메뉴 명`

**판매량 컬럼 허용값:**
- `판매량(개)`
- `판매량 (개)`

**허용하지 않음:**
- `판매량`
- `수량`
- `판매개수`
- 기타 임의 헤더

### 사용 컬럼 정책

현재 메뉴판매량 업로드에서 **실제로 사용하는 컬럼:**
- `메뉴명`
- `판매량(개)`

**현재 사용하지 않는 컬럼:**
- `카테고리` → **판매량 분류나 집계 기준으로 사용하지 않음**
- `매출액(원)`
- `매출 비중`

분류는 **내부 분류 엔진의 결과를 기준**으로 한다.

### rawRows와 validRows 구분 정책

**rawRows:**
- 엑셀/CSV 원본 2차원 배열
- 기간 탐지 대상
- 헤더 탐지 대상
- `validateSalesFile(rawRows)` 검증에 사용
- `classifyAndPrepare(validRows, ...)` 입력에는 사용 금지

**validRows:**
- 파일 로더에서 검증 통과한 객체형 데이터
- 메뉴명, 판매량만 포함
- `classifyAndPrepare()` 입력으로 사용
- `classifyAndPrepare()` 출력으로 사용

**금지:**
```javascript
// ❌ 금지
validateSalesFile(rows)
```

**허용:**
```javascript
// ✅ 허용
validateSalesFile(rawRows)
```

### 저장 및 분류 대상 정책

판매량 저장과 분류는 원본 `rawRows`를 직접 쓰지 않는다.

**반드시 아래 흐름을 따른다:**

```
validateSalesFile(rawRows)
  ↓
result.validRows
  ↓
transformedRows
  ↓
currentParseResult.validRows
  ↓
classifyAndPrepare(validRows, year, month)
  ↓
classifiedRows + groupedIssues
  ↓
saveSalesUpload(...)
```

즉, **검증을 통과한 유효 행만** 분류와 저장 대상으로 사용한다.

### 데이터 행 검증 정책

**메뉴명:**
- 빈칸 차단
- 공백만 있는 값 차단

**판매량:**
- 빈칸 차단
- 숫자가 아닌 값 차단
- 소수 차단
- 음수 차단
- 0은 허용

### 업로드 차단 기준

- `invalidRows`가 **1건이라도 있으면 전체 업로드 차단**
- 일부 정상 행만 저장하는 방식 금지

### 업로드 UX 정책

**파일 선택 후:**
- 즉시 DB 저장하지 않는다.
- 먼저 검증 결과 화면을 보여준다.

**검증 결과 화면 표시 항목:**
- 기간
- 헤더 확인
- 데이터 행 수

**저장 시점:**
- 사용자가 저장 버튼을 눌렀을 때만 DB 저장

### 같은 월 중복 업로드

- 동일한 `year + month` 데이터가 이미 있으면 저장 차단
- 중복 안내 UI 표시

### 기존 월 데이터 삭제

- 사용자가 명시적으로 삭제 버튼 클릭
- 재확인 모달 표시
- 삭제 완료 후 자동 저장 금지
- 사용자가 다시 저장 버튼을 눌러야 저장됨

## 메뉴 분류 파이프라인 정책

저장 직전 판매량 메뉴명은 아래 순서로 처리한다:

```
rawMenuName
  → normalizeMenuName()
  → mapAlias()
  → matchRule()
  → classifyMenuStatus()
```

### 상태값 정책

| 상태 | 의미 | 통계 포함 | 관리 필요 |
|------|------|----------|----------|
| classified | 정상 분류 완료 | ✅ | ❌ |
| unclassified | 분류 실패 | ❌ | ✅ |
| excluded | 품목 제외 확정 | ❌ | ❌ |

### MS-6 후속 보완: 업로드 시 DB 기준 자동 반영 정책

**목적**: 메뉴판매량 설정(MS-7)에서 등록한 기준 데이터를 새로운 업로드에 자동으로 적용하여 분류 정확도를 높인다.

**자동 반영 대상**:
- `ref_sales_aliases` — 메뉴명 정규화 후 별칭 매핑 (예: 테스트 → 테스트1 → 테스트12)
- `sales_rules` — 정규화된 메뉴명의 카테고리/그룹 분류 규칙
- `ref_excluded` — 판매량 집계 제외 대상 메뉴

**처리 순서**:
1. 업로드 파일 검증 및 파싱
2. **새 업로드마다 DB 기준 데이터 동적 로드** (`createMenuSalesClassifier()`)
3. 정규화 → DB 별칭 매핑 → DB 규칙 매칭 → 제외 판정
4. 분류 완료 후 저장

**메모리 상수와 DB 기준의 우선도**:
- DB 규칙이 있으면 DB 규칙 적용 (우선도 높음)
- DB 규칙이 없으면 메모리 상수 적용 (fallback)
- 메모리 별칭과 DB 별칭이 모두 있으면 DB 별칭 우선

**이점**:
- 메뉴판매량 설정에서 한 번 등록하면 이후 모든 업로드에 자동 반영
- 메뉴 변경, 신규 메뉴 추가 시 설정 탭에서만 관리
- 업로드 시마다 최신 기준이 적용됨 (코드 수정 불필요)

## 메뉴판매량 순위 탭 정책

### 목적

업로드된 월별 판매량 데이터에서 **메뉴별 판매량 순위와 카테고리별 순위를 확인**하는 화면이다.

### 데이터 기준

- **원본:** `sales_rows`
- **필터:** `status = 'classified'`만 집계
- **제외:** `unclassified`, `excluded`는 집계 제외

### 기본 조회 단위

**기본 단위: `groupName` 기준 합산**

예:
- `샘스테이크L`
- `샘스테이크R`

이 둘은 기본 순위에서:
- `샘스테이크`

로 합산한다.

### 상세 보기 옵션

기본은 `groupName` 기준이지만, 사용자는 필요 시 `detailName` 기준으로 세부 순위를 볼 수 있다.

| 보기 방식 | 의미 |
|----------|------|
| 메뉴 통합 순위 | groupName 기준 |
| 상세 메뉴 순위 | detailName 기준 |

### 필수 필터

- **조회 월 선택**
- **카테고리 선택**
- **메뉴명 검색**

### 카테고리 예시

```
전체
피자
1인피자
사이드
사이드(소스)
세트메뉴
엣지&도우
추가토핑
하프앤하프
음료
```

### 기본 정렬

- 판매량 내림차순
- 동일 판매량이면 메뉴명 가나다순

### 표 기본 컬럼

| 컬럼 | 설명 |
|------|------|
| 순위 | 판매량 순위 |
| 카테고리 | 내부 분류 카테고리 |
| 메뉴명 | groupName 또는 detailName |
| 판매량 | 합산 판매량 |
| 판매 비중 | 선택 월 전체 classified 판매량 대비 비율 |

### 순위 화면에서 하지 않을 것

- ❌ 집계 결과 DB 저장
- ❌ unclassified 포함
- ❌ excluded 포함
- ❌ 원본 엑셀 카테고리 기준 집계

## 메뉴판매량 비교 탭 정책

### 목적

서로 다른 월 또는 기간의 메뉴 판매량을 비교하여 **증감 메뉴와 성장률을 확인**하는 화면이다.

### 데이터 기준

- **원본:** `sales_rows`
- **필터:** `status = 'classified'`만 비교
- **기본 단위:** `groupName` 기준

### 비교 방식

**기본 비교:**
- 기준 월 A
- 비교 월 B

예:
- 2026년 4월 vs 2026년 5월

### 비교 계산식

| 항목 | 계산식 |
|------|--------|
| 증감량 | 비교월 판매량 - 기준월 판매량 |
| 증감률 | (비교월 판매량 - 기준월 판매량) / 기준월 판매량 × 100 |

### 예외 처리

**기준월 판매량 0, 비교월 판매량 > 0:**
- 증감률은 % 계산 대신 **신규 표시**

**기준월 판매량 > 0, 비교월 판매량 0:**
- 표에는 **-100%**, 별도 라벨로 **판매 중단** 표시

### 비교 화면 기본 컬럼

| 컬럼 | 설명 |
|------|------|
| 카테고리 | 분류 카테고리 |
| 메뉴명 | groupName |
| 기준월 판매량 | A월 |
| 비교월 판매량 | B월 |
| 증감량 | B-A |
| 증감률 | 변화율 |

### 추가 보기

- 판매량 증가 TOP
- 판매량 감소 TOP
- 신규 등장 메뉴
- 비교월 미판매 메뉴

### 차트 정책

- 초기 구현은 **표 중심**
- 차트는 **후속 구현** 가능
- 차트 추가 시:
  - 막대 차트: 월별 판매량 비교
  - 증감 차트: 증가/감소 메뉴 시각화

### 비교 화면에서 하지 않을 것

- ❌ 비교 결과 DB 저장
- ❌ unclassified 포함
- ❌ excluded 포함
- ❌ 사용자가 선택하지 않은 과거 월 자동 비교

## 메뉴미매칭 관리 탭 정책

### 목적

분류되지 않은 판매량 메뉴를 확인하고, **관리자가 검토하여 향후 자동 분류되도록 보정**하는 화면이다.

### 데이터 기준

- **원본:** `menu_sales_issues`
- **기본 대상:**
  - `issueType = 'unmatched'`
  - `status = 'open'`

### 이슈 그룹 기준

현재 이슈는 업로드 저장 시 아래 기준으로 그룹화되어 저장된다:

```
같은 월
+ 같은 issueType
+ 같은 normalizedMenuName
```

### 표시 정보

| 항목 | 설명 |
|------|------|
| 월 | year / month |
| 이슈 유형 | unmatched |
| 정규화 메뉴명 | normalizedMenuName |
| 대표 원본 메뉴명 | representativeRawMenuName |
| 영향 행 수 | affectedRowCount |
| 판매량 합계 | totalQuantity |
| 연결 행 | relatedSalesRowIds |
| 상태 | open |

### 관리자 조치 정책

미매칭 항목은 아래 중 하나로 처리할 수 있어야 한다:

#### A. 별칭 매핑 등록

예: `치즈가루` → `파마산`

#### B. 신규 분류 규칙 등록

예: 새로운 메뉴명을 특정 `category / groupName / detailName`에 매핑

#### C. 품목 제외 등록

예: 통계 집계 대상이 아닌 항목

### 조치 후 반영 정책

미매칭 관리에서 사용자가 처리를 확정하면 **아래 2가지가 함께 반영**된다:

**현재 업로드 데이터:**
- 해당 issue의 `relatedSalesRowIds`를 즉시 재처리
- `sales_rows.status` 업데이트
- issue 상태를 `resolved`로 변경

**미래 업로드 데이터:**
- 별칭 / 규칙 / 제외 설정을 저장하여
- 다음 업로드부터 자동 처리

### 미매칭 관리에서 하지 않을 것

- ❌ 관리자 확인 없이 자동 매핑
- ❌ 유사 문자열 기반 추정 분류
- ❌ 과거 전체 월 자동 재분류
- ❌ 기존 집계 결과를 강제로 재작성

## 메뉴판매량 설정 탭 정책

### 목적

판매량 분류와 관련된 기준 데이터를 관리하는 화면이다.

### 주요 관리 대상

#### 1. 별칭 관리

특정 원본/정규화 메뉴명을 표준명으로 변환

예:
- `뉴더블 피자` → `뉴더블치즈피자`
- `파마산추가` → `파마산`

#### 2. 품목 제외 관리

통계에서 제외할 메뉴명 관리

제외 처리된 메뉴는 `excluded`로 판정 가능

## 메뉴 그룹 매핑 규칙

`menu-sales-ms9-rules.js` 및 DB `sales_rules`에 등록되는 분류 규칙은 아래 원칙에 따라 관리한다.

### 매핑 원칙

#### 1. 동일 메뉴 다중 표기 통합

같은 메뉴의 여러 표기(L/R/기본명/피자/변경/추가)는 모두 동일 `groupName`으로 합산한다.

| detailName 패턴 | 순위 화면 표시 |
|----------------|--------------|
| `groupName + L` | 단독 행 (L) |
| `groupName + R` | 단독 행 (R) |
| 그 외 | `groupName + 기타`로 합산 |

#### 2. 메뉴명 변경 정책

메뉴명 변경 시 신구 두 이름 모두 동일 `groupName`으로 매핑되도록 양쪽 규칙을 등록한다.
- **신메뉴명을 정식 `groupName`으로 사용**한다.
- 구메뉴명도 같은 `groupName`으로 매핑되도록 별도 규칙 추가.
- 과거 데이터와 신규 데이터가 같은 그룹으로 자동 집계된다.

#### 3. 신메뉴 사전 등록 정책

신메뉴는 출시 전 6종 변형(L / R / 기본명 / 피자 / 변경 / 추가)을 사전 등록한다. 첫 판매 발생 시점부터 자동 분류된다.

#### 4. 데이터 오타·별칭 처리

엑셀 원본의 오타나 약어 표기는 같은 `groupName`으로 매핑되는 별도 규칙으로 처리하며, alias 테이블이 아닌 규칙 테이블에 추가한다(분류 규칙 우선 적용 원칙).

#### 5. PCS 카운트 분리 (사이드)

사이드 카테고리는 PCS 단위 메뉴가 많아 카운트별로 단독 행을 표시한다.

- `detailName === groupName + '{숫자}' + 'pcs'` 패턴이면 카운트별 단독 행
- 예: `핫윙4pcs`, `핫윙20pcs`, `치킨바사삭6pcs`, `치킨바사삭12pcs`
- 한 그룹에 여러 카운트가 공존하면 카운트별로 분리되어 표시 (오름차순)

#### 6. 콘코울슬로 콤보 분할 (사이드)

`+콘코울슬로`가 포함된 콤보 메뉴는 분류 단계에서 자동으로 두 그룹에 집계된다.

- 원본 행: 주메뉴 그룹의 해당 pcs 행으로 분류 (예: `치킨바사삭6pcs`)
- 가상 행: 동일 수량으로 콘코울슬로 그룹에도 추가 (`menu-sales-classify.js`에서 자동 처리)
- 결과: 100개 콤보 판매 시 치킨바사삭 그룹 +100, 콘코울슬로 그룹 +100
- 이유: 콤보 판매에 실제 두 품목이 모두 포함되어 메뉴개발 관점에서 양쪽 집계가 필요

### 품목제외 카테고리 규칙 (2026-05 기준)

**총 1개 그룹 / 20개 규칙**

#### 처리 방식

품목제외는 **규칙 매칭만으로 자동 `status='excluded'` 처리**된다. `status-classifier.js`에서 `matchedRule.category === '품목제외'`인 경우 분류 결과를 `excluded`로 확정한다.

- 통계/순위/비교 화면에서 자동 제외 (`status='excluded'`)
- 미매칭 관리 UI에도 노출되지 않음 (이미 분류 완료된 상태)
- 기존 `ref_excluded` DB 메커니즘과 병행 운영 (DB 우선)

#### 상태 판정 우선순위 (`status-classifier.js`)

```
1. exclusionDecision === 'excluded'       → excluded (DB ref_excluded)
2. exclusionDecision === 'exclude_candidate' → unclassified
3a. matchedRule.category === '품목제외'    → excluded (규칙 기반)  ← 신규
3b. matchedRule 존재                       → classified
4. 그 외                                   → unclassified
```

리뷰이벤트, 사장님 금액 할인, 쿠폰/할인,

피클/갈릭디핑소스/핫소스 제외, 갈릭디핑/핫소스 제외,
