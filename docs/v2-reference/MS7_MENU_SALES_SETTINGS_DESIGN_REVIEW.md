### 구현된 기능

#### 1.1 탭 기반 UI (3개 탭)
- **별칭 관리** (`alias`)
  - 데이터 소스: `ref_sales_aliases` 테이블
  - 제공 기능: 목록 조회, 활성/비활성 토글, 삭제, 추가
  - 테이블 컬럼: 원본/정규화명, 매핑 대상, 상태, 조작

- **품목 제외** (`excluded`)
  - 데이터 소스: `ref_excluded` 테이블
  - 제공 기능: 목록 조회, 삭제, 추가
  - 테이블 컬럼: 메뉴명, 상태(제외), 조작

- **분류 규칙** (`rule`)
  - 데이터 소스: `sales_rules` 테이블
  - 제공 기능: 목록 조회, 비활성화, 추가
  - 테이블 컬럼: 메뉴명, 카테고리, 그룹명, 상태, 조작

#### 1.2 데이터 저장 정책
- **저장 방식**: 즉시 저장 (모달 "저장" 버튼 클릭)
- **활성/비활성**: `enable` 플래그로 관리
- **삭제**: 실제 삭제 (hard delete)
- **중복 방지**: 로직 미포함 (추가 시 중복 체크 없음)

#### 1.3 UI 패턴
- 모달 패널 기반 입력 폼
- 백드롭 클릭 시 닫힘
- 토스트 메시지로 결과 알림

## 3. 설계 검토 항목

### ✅ 현재 올바른 설계 결정

1. **Lazy Load 탭 구조**
   - 메뉴판매량 모듈 시작 시는 로드하지 않음
   - 사용자가 "메뉴판매량 설정" 탭 클릭 시에만 초기화
   - 이점: 초기 로드 성능 영향 최소화

2. **활성/비활성 관리**
   - 삭제보다 비활성화 우선 (정책 준수)
   - 기존 이력 보존 가능
   - 재활성화 가능

3. **모달 기반 입력**
   - 화면 내 inlineform이 아닌 모달 방식
   - 사용자 실수 방지 (명시적 저장)

4. **DB 테이블 통합**
   - 3개 탭이 각각 독립 테이블 관리
   - 명확한 책임 분리

### ⚠️ 검토 필요한 설계 이슈

#### 3.1 중복 등록 방지 미구현
**현재 상태**: 중복 체크 없음

**예상 문제**:
- 사용자가 실수로 동일 별칭을 여러 번 등록 가능
- 동일 규칙을 여러 번 등록 가능
- 동일 제외 메뉴를 여러 번 등록 가능

**정책**: MENU_SALES_DETAILED_POLICY.md에 정의
> **중복 방지:**
> - 동일 alias 중복 등록 방지
> - 동일 제외 메뉴 중복 등록 방지
> - 동일 규칙 중복 등록 방지

**권장 해결책**:
```javascript
// 별칭 추가 시
const existing = aliases.find(a => a.rawName === rawName);
if (existing) {
  toast.warn('이미 등록된 별칭입니다.');
  return;
}

// 규칙 추가 시
const existing = rules.find(r => r.rawMenuName === rawMenuName);
if (existing) {
  toast.warn('이미 등록된 규칙입니다.');
  return;
}

// 제외 추가 시
const existing = excluded.find(e => e.menuName === menuName);
if (existing) {
  toast.warn('이미 등록된 제외 항목입니다.');
  return;
}
```

#### 3.2 데이터 검증 미흡
**현재 상태**: 최소 검증 (빈값만 확인)

**예상 문제**:
- 특수문자, 매우 긴 문자열 등 입력 제한 없음
- 카테고리 값이 하드코딩된 옵션 목록(select)으로만 제한되어 있음 (좋음)
- 하지만 rawMenuName, mappedName, groupName은 자유 입력

**권장 개선**:
```javascript
function validateAliasInput(rawName, mappedName) {
  if (!rawName || !mappedName) return { ok: false, msg: '필수항목 누락' };
  if (rawName.length > 50) return { ok: false, msg: '원본명이 너무 깁니다 (50자 이하)' };
  if (mappedName.length > 50) return { ok: false, msg: '매핑명이 너무 깁니다 (50자 이하)' };
  if (rawName === mappedName) return { ok: false, msg: '원본명과 매핑명이 같을 수 없습니다' };
  return { ok: true };
}
```

#### 3.3 수정(Edit) 기능 미구현
**현재 상태**: 
- 별칭: 활성/비활성화만 가능 (전체 수정 불가)
- 규칙: 비활성화만 가능 (수정 불가)
- 제외: 삭제만 가능 (수정 불가)

**예상 문제**:
- 오입력한 데이터를 수정하려면 삭제 후 다시 추가해야 함
- 관리자 실수 후 복구 어려움

**정책 확인**: MENU_SALES_DETAILED_POLICY.md
> **활성/비활성 관리:**
> - 삭제보다 **비활성화 우선**

> **변경 영향도:**
> - 설정 변경이 **과거 데이터 전체에 자동 반영되지는 않음**
> - 과거 데이터 재분류는 **별도 명시 실행이 필요**

**권장 판단**:
정책에서 "수정" 기능을 명시하지 않았으므로:
- 삭제 후 재등록하는 방식도 정책 준수 가능
- 하지만 사용성 관점에서 "수정" 기능 추가 권장

#### 3.4 삭제 전 확인 UI 미구현
**현재 상태**: 삭제 버튼 클릭 시 바로 삭제 (확인 없음)

**예상 문제**: 사용자 실수로 중요 데이터 의도치 않게 삭제

**권장 개선**: 인라인 확인 모달 추가
```javascript
// 삭제 버튼 클릭 시
function confirmDelete(type, id, name) {
  showConfirmModal(`정말로 삭제하시겠습니까? (${name})`, async () => {
    await deleteItem(type, id);
  });
}
```

#### 3.5 대량 관리 기능 미포함
**현재 상태**: UI를 통한 개별 추가/삭제만 가능

**예상 문제**: 50개 이상의 규칙/별칭 관리 시 진입 장벽 높음

**권장 개선** (후순위):

- 정렬 / 검색 기능
