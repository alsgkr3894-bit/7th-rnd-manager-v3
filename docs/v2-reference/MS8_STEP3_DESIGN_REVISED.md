## 2. B-1 '이슈 수 일치' 최종 처리 방안

### 문제 분석

B-1: "화면 count vs 실제 menu_sales_issues 비교"

**성격 판정:**
- ❌ **데이터 정합성 검증 아님**
- ✅ **UI 표시값 재확인에 가까움**

**이유:**
```
menu_sales_issues는 업로드 시 저장된 데이터
  ↓
저장 시점에 이미 정합성이 보장됨 (원자적 트랜잭션)
  ↓
월별 이슈 count 조회 = 저장된 값 재확인일 뿐
  ↓
이는 데이터 손상을 감지하는 정합성 검증이 아님
```

**정상 상황 예:**
- 월별 이슈 조회: 12건
- menu_sales_issues 실제: 12건
- → **정상** (손상 없음)

**비정상 상황 예:**
- orphan row를 가리키는 이슈 존재
- → **정상 감지하는 항목: B-2**

### 최종 결정: **C번 선택**

✅ **Step 3 최소 구현안에서는 제외**
- 이유: 정합성 검증 기준 미충족

✅ **브라우저 회귀 테스트 항목으로만 관리**
- 검증 버튼 클릭 시 issue 모달이 정상 열리는지
- 토스트에 표시되는 issue count가 화면 값과 일치하는지
- 이는 UI 기능 테스트 범위

## 3. 경량 검증 vs 정밀 검증 분리표

### 3.1 DB 조회 비용 분석

| 항목 | 로직 | 조회 비용 | 개수 |
|------|------|---------|------|
| A-1 | fileId별 sales_rows 개수 | 낮음 (인덱스) | 1회 |
| A-2 | fileId별 row count = 0 | 낮음 (인덱스) | 1회 |
| D-1 | year/month 중복 | 낮음 (인덱스) | 1회 |
| E-1 | status별 groupBy | 낮음 (전체) | 1회 |
| **B-2** | **relatedSalesRowIds 각 row getById** | **높음 (N회)** | **N회** |
| **C-1** | **linkedFileId 기준 log 조회 + sales_files 존재 확인** | **중간 (2회)** | **1회** |
| **C-2** | **linkedFileId 기준 deleted log 중복** | 낮음 (인덱스) | 1회 |

### 3.2 최종 분리 기준

#### 🟩 경량 검증 (카드 렌더링 시 즉시 실행)

```javascript
경량 검증(year, month) {
  // 각 조회: 1~2회, 인덱스 기반
  // 총 DB 조회: ~4회
  // 예상 시간: 10~50ms
  
  ✓ A-1: 파일-행 개수 일치
  ✓ A-2: 파일 orphan
  ✓ D-1: 년월 중복
  ✓ E-1: 상태 분포 합계
}
```

**특징:**
- sales_rows 전체 1회 조회 + index 3회 → count만 비교
- row 상세 데이터 조회 없음
- relatedSalesRowIds 확인 없음
- 월별 데이터 렌더링 중에 실행 가능

#### 🔵 정밀 검증 (상세 보기 모달에서 온 디맨드 실행)

```javascript
정밀 검증(year, month) {
  // 조회: N회 (B-2), 2회 (C-1)
  // 총 DB 조회: 3+N회 (N=orphan issue 개수)
  // 예상 시간: 50~500ms (이슈 개수에 따라)
  
  ✓ B-2: orphan 이슈 (relatedSalesRowIds 확인)
  ✓ C-1: 삭제 일관성
}
```

**특징:**
- 사용자가 "상세 보기" 클릭 후 모달에서 실행
- N개의 orphan 이슈 각각에 대해 row 존재 확인
- 선택적이므로 성능 영향 최소화

### 3.3 분리 구조

```
월별 카드 렌더링
  ├─ 경량 검증 4개 실행 (동기 또는 빠른 비동기)
  │  ├─ A-1 체크
  │  ├─ A-2 체크
  │  ├─ D-1 체크
  │  └─ E-1 체크
  │  └─ 배지 렌더링 (✅/⚠️/❌)
  │
  └─ "상세 보기" 버튼 제공
      └─ 클릭 시 모달 열기
          └─ 정밀 검증 2개 실행 (온 디맨드)
              ├─ B-2 체크
              └─ C-1 체크
              └─ 상세 정보 모달 표시
```

## 4. 삭제 로그 검증 기준 재정의

### 4.1 deleteSalesMonth() 정책 재분석

```javascript
// 삭제 프로세스
deleteSalesMonth(year, month) {
  1. sales_files: hard delete (완전 삭제)
  2. sales_rows: hard delete (완전 삭제, fileId 기준)
  3. menu_sales_issues: hard delete (완전 삭제, year_month 기준)
  4. upload_log: soft delete only!
     - deleted = true
     - deletedAt = new Date()
     - 레코드 자체는 유지
}
```

### 4.2 C-1/C-2 검증 항목 재정의

#### ❌ 기존 C-1 (폐기)
- "deleted=true인데 sales_files 존재"
- **문제**: deleteSalesMonth()에서 hard delete하므로 불가능한 조건
- **결과**: 항상 거짓이므로 검증 가치 없음

#### ❌ 기존 C-2 (폐기)
- "linkedFileId가 deleted=true 2건 이상"
- **문제**: upload_log는 linkedFileId별로 정확히 1건만 존재 (unique 구조)
- **이유**: 재업로드 시 새 fileId 생성되므로 같은 linkedFileId는 1번만 삭제됨
- **결과**: 구조상 불가능한 조건

### 4.3 개선된 C-1 (정의 변경)

#### ✅ 새로운 C-1: "삭제 일관성" (정밀 검증)

**검증 내용:**
```
upload_log.linkedFileId가 sales_files에 없으면서
deleted=false인 경우 → 오류
```

**판단 로직:**
```javascript
C-1 검증(year, month):
  1. 해당 월 upload_log 조회
  2. linkedFileId가 sales_files에 존재하는지 확인
  3. 없으면서 deleted=false → 오류
     (의미: 파일 기록은 남아있는데 실제 데이터는 없음)
```

**정상 상황:**
```
✓ deleted=false + sales_files 존재 (활성 파일)
✓ deleted=true + sales_files 없음 (정상 삭제됨)
```

**비정상 상황:**
```
✗ deleted=false + sales_files 없음 
  → 파일 기록은 활성인데 실제 데이터는 삭제됨 (일관성 깨짐)
```

### 4.4 C-2 폐기 및 대체

#### ❌ C-2 제거 사유
- 구조상 불가능한 조건 (linkedFileId unique 제약)

#### ✅ 대체 항목 검토
| 항목 | 내용 | 필요성 |
|------|------|--------|
| **F-1** | 동일 year/month 다중 재업로드 기록 | 중간 |
| 확장 | 같은 월을 여러 번 업로드/삭제한 이력 추적 | 향후 |

**현재 결론**: C-2는 제외하고, C-1 (삭제 일관성)만 정밀 검증에 포함

## 5. 수정된 최소 구현안

### 5.1 검증 항목 최종 정의

#### 🟢 최소 구현 항목: 6개 (A-1, A-2, B-2, C-1, D-1, E-1)

| 구분 | # | 항목 | 중요도 | 타입 | 설명 |
|------|---|------|--------|------|------|
| **경량** | A-1 | 파일-행 개수 일치 | 🔴 높음 | 경량 | fileId별 sales_rows count = totalRows |
| **경량** | A-2 | 파일 orphan | 🔴 높음 | 경량 | fileId로 연결된 row 0건 |
| **정밀** | B-2 | orphan 이슈 | 🔴 높음 | 정밀 | relatedSalesRowIds 각 row 존재 확인 |
| **정밀** | C-1 | 삭제 일관성 | 🔴 높음 | 정밀 | linkedFileId 존재 vs deleted 플래그 일관성 |
| **경량** | D-1 | 년월 중복 | 🔴 높음 | 경량 | year/month 중복 존재 여부 |
| **경량** | E-1 | 상태 분포 합계 | 🟡 낮음 | 경량 | classified+unclassified+excluded = totalRows |

### 5.2 제외된 항목

| 항목 | 이유 |
|------|------|
| **B-1** | UI 재확인에 가까움 (정합성 검증 아님) → 브라우저 회귀 테스트로만 관리 |
| **C-2** | 구조상 불가능 (linkedFileId unique 제약) → 제외 |

### 5.3 구현 함수 시그니처
