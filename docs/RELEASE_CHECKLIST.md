# 7rnd v3 배포 전 검증 체크리스트

이 문서는 배포 또는 실제 업무 사용 전 마지막 확인 순서를 정의한다. 기능 코드, 계산식, 저장/삭제, 업로드/다운로드, DB 구조를 바꾸지 않고 검증만 수행한다.

## 1. 시작 전 기준 확인

- [ ] 현재 브랜치가 배포 대상 브랜치인지 확인한다.
- [ ] `git status --short`가 비어 있는지 확인한다.
- [ ] 최신 커밋 해시와 배포 대상 커밋 해시를 기록한다.
- [ ] `package.json`, lock 파일, `.env`, DB 파일 변경이 없는지 확인한다.
- [ ] 실데이터 QA 전에 설정/백업에서 JSON 백업 파일을 만든다.
- [ ] 삭제/복원 검증은 운영 데이터가 아닌 복구 가능한 데이터로만 수행한다.

## 2. 자동 검증 순서

아래 순서대로 실행하고 결과를 `docs/QA_RESULT_TEMPLATE.md`에 기록한다.

1. `git status --short`
2. `npm run lint`
3. `npm test`
4. `npm run qa:smoke`
5. `npm run build`

기대 결과:

- lint: 경고와 오류가 없어야 한다.
- test: 전체 Jest 테스트가 통과해야 한다.
- qa:smoke: 702px viewport 기준 22/22 통과해야 한다.
- build: Next production build가 성공해야 한다.

## 3. 실데이터 QA 순서

세부 체크리스트는 `docs/QA_CHECKLIST.md`를 따른다.

1. 백업 파일 생성
2. 읽기 전용 화면 확인
3. 검색/필터/정렬/페이지네이션 확인
4. 실제 엑셀 업로드 확인
5. 업로드 결과 화면값과 원본 파일 대조
6. 다운로드 파일 생성 확인
7. 다운로드 파일을 실제 Excel 앱에서 열어 확인
8. 저장/수정/삭제는 테스트 데이터에서만 확인
9. 원가 계산 결과를 기준표와 비교
10. 백업/복원은 복구 가능한 데이터로만 확인

## 4. 문제 발생 시 중단 기준

아래 상황에서는 배포 또는 실사용 전환을 중단한다.

- build 실패
- smoke QA 실패
- 주요 라우트에서 빈 화면 또는 Application error 발생
- 실제 업무 파일 업로드 후 저장 결과가 원본과 다름
- 다운로드 파일의 컬럼 순서나 한글이 깨짐
- 원가 계산 결과가 기준표와 불일치
- 삭제/복원 결과가 예상과 다름
- DB 스키마, API, 라우터, 저장 구조 변경이 필요해 보임

## 5. 되돌리는 방법

코드를 되돌려야 할 때는 destructive 명령을 우선 사용하지 않는다.

- 아직 push 전이면 담당자에게 문제 커밋 해시와 검증 실패 내용을 보고한다.
- 이미 커밋된 변경을 되돌릴 때는 `git revert <commit>` 방식으로 되돌림 커밋을 만든다.
- 데이터 문제는 코드 revert 전에 설정/복원에서 QA 전 백업 파일로 복원 가능한지 확인한다.
- 운영 데이터 삭제 또는 마이그레이션이 필요한 경우 즉시 중단하고 별도 복구 계획을 작성한다.
